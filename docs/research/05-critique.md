# Adversarial Critique — Dish Development Workbench (DESIGN.md v0.2)

Devil's-advocate review. No overall verdict (the Reviewer renders that). Attacks the strongest version of the design, and audits the scouts where they were soft or wrong. Calibration: portfolio flagship for FAANG SWE/systems internship review; market defensibility is a non-goal.

## Top blocking concerns

**[High] The eval measures the wrong thing — the hero hypothesis is orthogonal to the product's actual edge.** §9.1 predicts grounding "helps correctness/provenance more than creativity," and the README headline (§17) stakes the pitch on *"does grounding actually beat a strong LLM?"* But the predicted correctness win is **USDA arithmetic and entity resolution** — which the design already routes through *deterministic functions* (§8.4), not the LLM-retrieval arm. The culinary scout makes this explicit (its §5): the clean win is "unglamorous determinism, not flavor science." So the ablation's most-likely positive result is a tautology — a calculator beats an LLM at arithmetic — and that win belongs to the generative/deterministic split (§7.2, §8.4), which the doc treats as *already settled*, not to grounding. Meanwhile the product-UX scout's own steelman (its §4) locates the real edge over ChatGPT in **durability/resumability + trustable numbers** — again the engineering shell, *not* flavor grounding. The hero artifact thus measures a hypothesis whose positive case is a tautology and whose contested case (flavor) is most-likely null, while *not measuring* the thing that actually differentiates the product. *De-risk:* make the eval's headline metrics process-level and sign-robust — hallucination/provenance rate, gate accept/edit/reject dynamics (§9.3 already lists these) — and demote "does grounding beat the LLM on flavor" to a supporting, openly-hedged result.

**[High] Live-literature retrieval is the timeline-killer, and it's wired as a hard dependency of the hero.** P0-7 (3-source hybrid grounding) sits in the wk2–6 spine with eight other P0 items (§15), yet a live-retrieval pipeline (query construction → fetch → citation extraction → ranking → fusion) is a multi-week subsystem on its own (architecture scout, High; market scout, feasibility). Worse, retrieval is a dependency of the grounded arm, which is a dependency of the ablation (the hero, §9.2). A retrieval slip cascades into the hero slipping. **Failure mode:** weeks lost to citation extraction, the eval ships as a thin 2-arm run on a tiny unvalidated benchmark, the gate machine is hand-waved — a half-working *everything* instead of one deep loop, violating the doc's own §2.4 principle via its own P0 list. *De-risk:* ship FlavorGraph+USDA/FoodOn as the grounded arm; move live literature to P1 layered on a working ablation (market rec 4, architecture rec 4).

**[Med] Buzzword density reads as resume-driven design.** Event-sourcing + CQRS + blackboard + hybrid 3-source RAG + streaming/interrupt + autonomy dial + 3-arm ablation, solo, one semester. A senior reviewer probes each label for depth; any aspirational one taints the rest. "Blackboard" (§8.3, glossary) is unearned — the pattern needs *multiple opportunistic knowledge sources* on a shared workspace; P0 has **one** generative agent (flavor reasoner, §8.4), so it's just a shared document. "Interruptible/checkpointed" overstates (no durable execution; architecture scout, Med). *De-risk:* drop "blackboard" and "CQRS"; claim only patterns you build to depth.

**[Med] No safety section for a system that proposes novel combinations to home cooks.** The culinary scout flags CRITICAL gaps (its §4): anaerobic oil infusions (botulism), undercooking, allergen-substitution blindness — and notes discovery mode (JTBD-3, §6.2-C) is *most* dangerous yet treated as lighter-weight P1. The doc has zero coverage. *De-risk (calibrated):* a narrow hard-coded blocklist (anaerobic infusions, poultry/egg min-temps, allergen check against `constraints.allergens`) + a documented disclaimer — not the full FDA-modeled rules engine the culinary scout implies, which would itself blow the scope budget (see "scouts too soft").

## The event-sourcing contradiction — resolved

The two scouts are praising and attacking **different referents under one word**, so there is no real conflict. The market scout rewards the *user-facing data-model story* — versioned, branchable, diffable, immutable history. The architecture scout attacks the *CQRS apparatus* — upcasters, snapshotting, projection rebuild (its High item). The git-style snapshot+diff chain with parent pointers + branch refs the architecture scout proposes delivers **100% of the data-model signal the market scout names** at a fraction of the build cost. You do not have to pick; you have to **stop calling the draft "event-sourced"** (§8.3) when it is a version chain. Honest version-chain claim = real signal; "event-sourced" label on a version chain = dishonest, which is strictly worse. The one place true event-sourcing *earns its keep* is the **move/gate event log** (§8.6, §11) — an append-only stream replayed for eval metrics. Locate event-sourcing there, relabel the draft, and both scouts are satisfied. Simplifying removes signal *only* if you keep the buzzword while shedding the build.

## Does the thesis survive a null result?

Partially — and only if the README is rewritten. The engineering spine that survives any ablation outcome is the gate state machine + version chain + deterministic split + the eval *apparatus and methodology itself* — none depend on flavor-pairing being true. The doc claims this posture (§2.4, §9.4: "report whatever is true"). **But §17 and §2.3 lead the pitch with the number** ("Built to test one question… does grounding actually beat…"). If the number is "no measurable flavor difference," the headline reads as "spent a semester confirming a contested 2011 hypothesis barely helps, plus a calculator beats an LLM at arithmetic." The fix is not new engineering — it is making *methodology + intellectual honesty* the headline and the number the footnote. As written, the doc's own README contradicts its own null-result-is-fine philosophy.

## Where the scouts are too soft or wrong

- **Market scout uncritically lists "event-sourced versioned draft" as a top-3 reward-worthy signal** (its §42–44) — repeating the doc's buzzword without asking whether it's event-sourced or just versioning. The architecture scout caught what the market scout rubber-stamped.
- **All scouts grade generously against "not a wrapper" but never ask whether clearing that floor equals a strong systems signal.** Strip retrieval + flavor science and the "systems" content is: a state machine + a version table + an SSE stream + a benchmark runner. Solid intern work — but the scouts don't test it against the *ceiling* the doc claims ("systems-engineering flagship," §1).
- **Architecture scout's 2-arm cut creates a new problem.** Dropping FlavorGraph-only (its rescope #7) removes the only arm that isolates *which* source helps. Under a null, 2 arms can't distinguish "grounding doesn't help" from "this retrieval impl is weak" — the 3-arm design has *more* diagnostic value precisely when the result is null. The cut trades build cost for interpretability of the hero.
- **Culinary scout's "safety gate as P0" ignores the scope budget** the architecture/feasibility scouts are screaming about. A botulism/temp/allergen rules engine is a multi-week build; for a non-shipped n=5 demo it is likely over-scoped versus a blocklist + disclaimer.

## Internal contradictions in DESIGN.md

- **"Food is substrate, engineering is the point" (§1) vs. the eval hero being a food-science hypothesis (§9.1).** If food were truly substrate, the hero artifact would measure orchestration/process quality — not whether 2011 flavor-pairing holds. You cannot call food "substrate" and stake the flagship deliverable on a food-science result. This is the central honesty crack.
- **FoodPuzzle "found live retrieval beats static-index RAG" stated as fact twice (§4, §8.5) and used as the §13 rationale for hybrid retrieval — but it is MISCHARACTERIZED** (market scout: it's a design contrast, no ablation; FoodPuzzle even caches offline). An architecture decision is justified by a fabricated empirical result a knowledgeable reviewer will catch.
- **USDA lumped under "flavor-science grounding" (§8.1, §8.5, §17) while §8.4 correctly scopes it to cost/nutrition** (culinary scout §2). USDA has zero flavor data, so the "3-source hybrid" for the *flavor* claim is really FlavorGraph + literature — calling it 3-source inflates it.
- **§8.3 labels the draft "event-sourced," yet its own acceptance criteria need only versioning** (architecture scout) — claim exceeds requirement.

## The 1–2 things to go deep on (and what to cut)

**Go deep on (1)** the move/gate state machine + proposal contract + git-style version chain — hand-rolled (architecture rec 2), the genuine differentiator; and **(2)** the eval harness + methodology, reframed so its hero metrics are sign-robust (provenance/hallucination/gate dynamics), decoupled from the live-retrieval dependency. **Cut/defer:** live-literature retrieval → P1; FlavorGraph-only arm → keep only if cheap (it aids null-interpretability, so don't cut it for build reasons — cut *literature* instead); the "blackboard" and "CQRS" labels → delete; streaming → scope explicitly to single-process in-session cancel; autonomy-dial → stays P1. Everything that makes this "not just Cursor-for-recipes" currently rests on the flavor-grounding leg — the weakest, possibly-null part. Shift the distinctiveness onto the **deterministic-numbers + durability** edge the UX scout already proved is the real advantage, and the project stops betting its identity on the shakiest science in the doc.
