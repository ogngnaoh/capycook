## License findings (summary)

All seven asset licenses in §10/Appendix B check out against primary sources — this table is in good shape and can ship largely as-is. Two refinements worth making: (1) note that FlavorGraph's README still carries a placeholder citation ("to appear"), reinforcing the doc's own "stale" flag rather than contradicting it; (2) for FoodKG, cite the canonical `foodkg/foodkg.github.io` (Haussmann et al., ISWC 2019) explicitly, since a same-named but unrelated `Gharibim/FoodKG` tool exists with no license at all — a careless citation could point a reviewer at the wrong, unlicensed repo. The "avoid gated Recipe1M via a lighter FoodOn+USDA KG" architectural decision is well-justified: the full FoodKG's dependency on Recipe1M+ is confirmed in its own ISWC paper.

## Competitive-read assessment

The corrected v0.2 positioning (§3.2) — that the real competitor is "ChatGPT + companions," not ChefGPT-style tools — holds up under direct inspection of all five named products. ChefGPT, FoodsGPT, and DishGen are unambiguously convenience/meal-planning tools (pantry photo or ingredient list → finished recipe → shopping list); none expose anything resembling a deterministic cost/nutrition split, versioned drafts, or a gated proposal loop. MyChefAI is a closer "coaching" analog (persona-based chefs for specific diets) but is still single-turn conversational, with no structured artifact or iteration history. Macaron is the one mischaracterization worth fixing: it's a general personal-AI-agent platform where cooking is a feature among many (bookings, calorie tracking, custom mini-apps), not a cooking-specific companion — DESIGN.md's grouping is reasonable but should say so explicitly to avoid overstating its food focus. The narrow claim in §3.2/§4 (no tool combines grounded reasoning + deterministic correctness + a versioned, human-gated co-development loop) is defensible against everything found in this scan, with the standard caveat that a universal "no tool exists" claim can't be proven, only not-falsified.

## Engineering-signal-to-FAANG read

What raises the signal: the human-gated move/proposal state machine, the event-sourced versioned draft, and the deterministic/generative split are all genuinely "not a wrapper" — a reviewer can see real state-machine design, a real data model, and a real architectural boundary even with the LLM stripped out, which is precisely the test DESIGN.md sets for itself (§2). The eval harness as hero artifact is a strong, differentiated choice: most portfolio LLM projects skip rigorous evaluation entirely, and a three-arm ablation with an honestly-reported null/modest result reads as senior-level intellectual honesty — rarer and more valuable to a reviewer than a flashy demo. The corrected, citation-checked related-work section (this report) also helps: reviewers who recognize FoodPuzzle/FoodSky/Chef Watson will notice if the claims about them are sloppy, and getting them right (or honestly hedged) is itself a credibility signal.

What lowers it: the scope is wide for solo semester work (orchestrator + streaming/interrupt transport + event-sourced store + hybrid retrieval + deterministic services + eval harness), and grounding/retrieval infrastructure is the most likely place effort balloons without proportional signal — a reviewer cares more about the gate/state-machine and the eval methodology than about retrieval sophistication. The "hybrid retrieval per FoodPuzzle's finding" framing, once corrected (this isn't an established empirical finding), should be reframed as the project's own hypothesis to test, which actually strengthens rather than weakens the engineering story — it makes the ablation a real contribution instead of a borrowed result.

## Feasibility (landscape view)

Comparable solo/small-team portfolio projects that ship in a semester typically deliver one of: (a) a well-scoped agentic loop with solid state management and a thin but real eval, or (b) a retrieval/RAG system with a clean benchmark — rarely both at full depth simultaneously, because each is its own multi-week subsystem. DESIGN.md's roadmap (§15) already anticipates this by sequencing the eval harness shell into v0 and deferring P1/P2 breadth, which is the right call. The riskiest scope item is §8.5's three-source hybrid retrieval (FlavorGraph + live literature retrieval + deterministic data) combined with a working three-arm ablation (§9.2) by week 6–10 — live literature retrieval with citation/provenance attachment is its own non-trivial subsystem (query formulation, source caching, citation extraction, dedup against FlavorGraph signal). A pragmatic semester cut: ship FlavorGraph + USDA/FoodOn as the grounded arm first, treat "live scholarly retrieval" as a stretch P1 addition layered onto an already-working ablation, rather than a P0 dependency the eval harness can't run without.

## Top 5 recommendations

1. **Fix the FoodPuzzle citation.** Change "found live scholarly retrieval beats static-index RAG" to something like "argues for web/scholarly retrieval over a static index by design (not a measured ablation)" in §4 and §8.5 — the current phrasing claims an empirical result the paper doesn't contain.
2. **Soften the FoodSky comparison framing.** "Beats much larger general models" → "beats ChatGPT-3.5 and a field of 7–13B open baselines" — only one larger closed model was actually in the comparison table found.
3. **Reconcile FoodSky's two number sets.** The arXiv preprint (~67%/66%, zero-shot) and the published Patterns headline (~83%/91%, full system) differ substantially; cite the published Cell Patterns figures (with a "secondary-source confirmed, primary text paywalled" note) since that's the version actually titled "Patterns 2025" in DESIGN.md.
4. **De-risk the hybrid-retrieval P0 dependency.** Sequence live scholarly retrieval as a P1 layer on top of a working FlavorGraph+USDA/FoodOn ablation, so the eval harness (the hero artifact) can ship and run even if literature retrieval slips.
5. **Disambiguate the Macaron and FoodKG citations.** Note Macaron's actual scope (general personal-AI agent, cooking as one feature) rather than implying it's cooking-dedicated, and cite the canonical `foodkg/foodkg.github.io` explicitly to avoid confusion with the unrelated, unlicensed `Gharibim/FoodKG` repo of the same name.
